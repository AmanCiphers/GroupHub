## Chapter 1 — Task Assignment Bug Fix

**Problem:** Members got `403 "Only the assigned member can update this task"` when trying to mark a task as done, even when they were the assignee.

**Root cause:** The `assigneeId` field on a fetched task is a *populated Mongoose document* (an object with `_id`, `fullName`, etc.), not a plain string. Calling `String(task.assigneeId)` produced `"[object Object]"` instead of the user ID string.

**Fix in `backend/src/services/task.service.js`:**

```js
// Before (broken):
if (String(task.assigneeId) !== String(userId))

// After (fixed):
const assigneeId = task.assigneeId ? String(task.assigneeId._id || task.assigneeId) : null
if (assigneeId !== String(userId))
```

**Lesson:** Always use `task.field._id || task.field` when comparing populated Mongoose ObjectId fields.

## Chapter 4 — Owner Review Actions

**Change:** Added `QuickStatusDropdown` for inline status changes and `ReviewActions` component for approve/request-changes workflow.

### QuickStatusDropdown

An inline `<select>` dropdown that replaces the static status badge in the owner's task detail modal. Lets the owner change task status without entering edit mode.

```jsx
function QuickStatusDropdown({ task, onUpdate }) {
  const handleChange = async (e) => {
    const payload = await apiFetch(`/api/v1/tasks/${task.id}`, {
      method: "PATCH",
      body: JSON.stringify({ status: e.target.value }),
    })
    onUpdate(payload.data.task)
  }

  return (
    <select value={task.status} onChange={handleChange}
      className={statusClass[task.status]}>
      {STATUS_OPTIONS.map(opt => (
        <option key={opt.value} value={opt.value}>{opt.label}</option>
      ))}
    </select>
  )
}
```

**Status options:**
```js
const STATUS_OPTIONS = [
  { value: "todo", label: "To Do" },
  { value: "in_progress", label: "In Progress" },
  { value: "needs_review", label: "Needs Review" },
  { value: "done", label: "Done" },
  { value: "cancelled", label: "Cancelled" },
]
```

### ReviewActions Component

Appears only when task status is `needs_review`. Two actions:

**Approve** → `PATCH /tasks/:taskId { status: "done", reviewNote: "" }`
- Marks the task as complete
- Progress is recalculated (moves toward 100%)

**Request Changes** → opens a textarea for feedback
- Calls `POST /tasks/:taskId/feedback { feedback: "..." }`
- This creates a subtask (see Chapter 5)
- Original task is closed (status → "done")

**File:** `app/projects/[id]/manage/page.jsx` — `QuickStatusDropdown` and `ReviewActions` components added before `TaskDetailModal`.

## Chapter 5 — Subtask Tree (Parent/Child)

**Change:** When the owner sends feedback on a `needs_review` task, instead of simply reverting status, a new child task is created linked to the original.

### Data Model

Added `parentId` to Task schema:

```js
parentId: {
  type: mongoose.Schema.Types.ObjectId,
  ref: "Task",
  default: null,
}
```

### Send Feedback Flow

```
Owner writes feedback on Task ABC-123 (status: needs_review)
  │
  ├── Task ABC-123 → status: "done" (parent)
  │     └── reviewNote: "Please fix the validation logic"
  │
  └── Task ABC-123-2 (child, parentId: ABC-123)
        ├── title: inherited from parent
        ├── description: "**Feedback from review:**\n{feedback}\n\n---\n**Original:**\n{description}"
        ├── assigneeId: same as parent
        ├── status: "todo"
        └── reviewNote: feedback text
```

## Appendix — Common Mistakes Encountered

| Mistake | Consequence | Prevention |
|---------|-------------|------------|
| Updated Mongoose enum but forgot Zod validator | API returned 400 "Invalid enum value" | Always update both validation layers |
| Compared `String(task.assigneeId)` directly when it was a populated object | 403 even for legitimate members | Use `String(task.assigneeId._id \|\| task.assigneeId)` |
| Used `Schema.Types.ObjectId` instead of `mongoose.Schema.Types.ObjectId` | ReferenceError: Schema is not defined | Always qualify with `mongoose.` in model files |
| Didn't restart dev servers | Changes not visible in browser | Kill and restart Next.js + nodemon after significant backend changes |
